# Publish FR Updates from TF Meeting Analysis

Post each Feature Request summary from a TF meeting analysis file as a comment on its corresponding GitHub issue.

## ROLE

Act as an automation assistant that reads the Issue Summaries section from a `/analyze-tf-meeting` output (after `/create-tf-issues` has resolved all `#TBD` placeholders) and posts each FR summary as a comment on the appropriate GitHub issue.

## PREREQUISITES

- `gh` CLI installed and authenticated with `repo` scope
- Target repository: `FinOps-Open-Cost-and-Usage-Spec/FOCUS_Spec`
- An analysis file previously generated by `/analyze-tf-meeting` with `#TBD` placeholders already resolved by `/create-tf-issues`

## INPUTS

**Analysis file** — automatically located by finding the most recent `*-Analysis.md` file in:
```
/Users/matthewcowsert/Obsidian/work/meetings/
```

Search recursively, sort by modification time, and use the newest file. Display the chosen file path and ask for confirmation before proceeding.

## PROCESS

### Step 1: Verify Prerequisites

Before parsing, check that the analysis file has no remaining `#TBD` placeholders in the Issue Summaries section. If any are found, warn the user:
```
Warning: Found {N} unresolved #TBD placeholders. Run /create-tf-issues first to resolve them.
```
Ask if the user wants to proceed anyway or abort.

### Step 2: Parse Issue Summaries

Read the analysis file and extract each FR summary block from inside the **second** fenced code block (the Issue Summaries section).

Each FR block is delimited by `---` separators and starts with:
```
## FR #{number} — {Title}
```

For each FR, extract:
- **Issue number**: The `#{number}` from the `## FR #{number}` heading
- **Comment body**: Everything from the `## Status Update - TF{#} – {date}` line through the end of the block (up to the next `---` or end of code fence)

The comment body does NOT include the `## FR #{number} — {Title}` line — that's just for identifying which issue to post to. The comment starts with the `## Status Update - TF{#} – {date}` heading. This heading format triggers the focus-automation bot to pin the comment to the top of the issue.

**CRITICAL — Heading level fix**: Before posting, verify the Status Update line starts with `## ` (h2). If the analysis file has `### Status Update` (h3), promote it to `## Status Update`. The automation bot only pins comments whose first line is an h2. This is a common issue because the analysis file nests Status Update under the FR title heading.

### Step 3: Display Summary

Print a table of FR updates to be posted:

```
## FR Updates to Publish

| # | FR | Title | Comment Length |
|---|-----|-------|---------------|
| 1 | #1888 | Remove Provider Name and Publisher Name... | 245 chars |
| 2 | #1060 | Establish FOCUS Metadata Specification... | 1,203 chars |
| 3 | #1059 | Establish FOCUS Attribute Specification... | 678 chars |
| 4 | #982 | Clarify Definitions of Effective Cost... | 2,105 chars |
| 5 | #958 | Add Non-Negotiated Discount Eligibility... | 89 chars |
```

State the total count and confirm: "Publishing {N} FR update comments to `FinOps-Open-Cost-and-Usage-Spec/FOCUS_Spec`. Proceeding."

### Step 4: Post Comments

For each FR summary, post the comment using `gh`:

```bash
gh issue comment {number} \
  --repo FinOps-Open-Cost-and-Usage-Spec/FOCUS_Spec \
  --body "{comment_body}"
```

Use a heredoc to pass the body to avoid shell escaping issues:

```bash
gh issue comment {number} \
  --repo FinOps-Open-Cost-and-Usage-Spec/FOCUS_Spec \
  --body "$(cat <<'EOF'
{comment_body}
EOF
)"
```

Post all comments, including those for skipped FRs (e.g., "TF3 — Skipped this meeting.").

Print each result as it completes:
```
Posted to #1888: Status Update - TF3 – 2026-02-06 (245 chars) [will be pinned]
Posted to #1060: Status Update - TF3 – 2026-02-06 (1,203 chars) [will be pinned]
```

If any post fails, report the error and continue with remaining items.

### Step 5: Report

Display a final summary:

```
## Results

| FR | Title | Status | URL |
|----|-------|--------|-----|
| #1888 | Remove Provider Name... | Posted | https://github.com/.../issues/1888#issuecomment-... |
| #1060 | Establish FOCUS Metadata... | Posted | https://github.com/.../issues/1060#issuecomment-... |

All {N} FR updates published successfully.
```

## ERROR HANDLING

- If `gh` is not authenticated, instruct user to run `gh auth login`
- If an FR issue number doesn't exist in the repo, report the error and skip that comment
- If the analysis file has no Issue Summaries section, report and abort
- If `#TBD` placeholders remain, warn but allow the user to proceed if they choose
- If a comment post fails, report the error and continue with remaining FRs

## USAGE

User can invoke this skill by:

- "Post the FR summaries to GitHub"
- "Publish the meeting updates to the feature requests"
- "Update the FRs with meeting notes"
