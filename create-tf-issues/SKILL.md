# Create GitHub Issues and Post PR Actions from TF Meeting Action Items

Create GitHub issues for standalone action items and post PR comments for PR-linked actions, both extracted by the `/analyze-tf-meeting` skill. Then update the analysis file by replacing `#TBD` placeholders with actual issue numbers.

## ROLE

Act as an automation assistant that takes the structured output from `/analyze-tf-meeting` and:
1. Creates GitHub issues for **Standalone Action Items** using the `gh` CLI (GraphQL API)
2. Posts comments on PRs for **PR Action Items** using the `gh` CLI
3. Patches the source file with real issue numbers

## PREREQUISITES

- `gh` CLI installed and authenticated with `repo` and `project` scopes
- Target repository: `FinOps-Open-Cost-and-Usage-Spec/FOCUS_Spec`
- An analysis file previously generated by `/analyze-tf-meeting`

## PROCESS

### Step 1: Read and Parse Action Items

Read the analysis file and extract action items from both subsections inside the first fenced code block (the Action Items section).

**1a. Parse Standalone Action Items** from the `### Standalone Action Items` subsection. Each AI is delimited by `---` separators and has this structure:

```
### [AI] {Title}

**Assignee(s):** @{github-handle}
**Parent Issue:** #{number} (or "N/A (description)")

**Detailed Description**
{paragraph}

**Definition of Done**
- [ ] {criterion}
```

For each standalone AI, extract:
- **Title**: The text after `### [AI] ` (keep the `[AI]` prefix)
- **Assignee(s)**: GitHub handle(s) after `@` (strip the `@`)
- **Parent Issue**: The `#number` -- used for the sub-issue relationship AND in the body
- **Body**: Everything from `**Detailed Description**` through the Definition of Done checklist
- **Due Date** (optional): Scan the Detailed Description for explicit date references matching patterns like `target: YYYY-MM-DD`, `target: Month DD`, `(target: YYYY-MM-DD)`, or `due date: ...`. If found, parse and use that date. If not found, use the default (current date + 7 days).

**1b. Parse PR Action Items** from the `### PR Action Items` subsection. Each PR Action is delimited by `---` separators and has this structure:

```
### [PR Action] {Title}

**Assignee(s):** @{github-handle}
**Target PR:** #{PR number}
**Comment Type:** {review-comment | general-comment}
**File/Line Hint:** {context or "N/A"}

**Description**
{paragraph}

**Scope**
- [ ] {item}
```

For each PR Action, extract:
- **Title**: The text after `### [PR Action] `
- **Assignee(s)**: GitHub handle(s) after `@` (strip the `@`)
- **Target PR**: The `#number` of the PR to comment on
- **Comment Type**: `review-comment` or `general-comment`
- **File/Line Hint**: Location context for review comments
- **Description**: The description paragraph
- **Scope**: The checklist items

Also extract from the file header:
- **Task Force**: The `TF-{#}` value from the `**Task Force:**` metadata line (e.g., `TF-3`). Used in Step 6 to set the Task Force project field.
- **Meeting Date**: For use in PR Action comment headers.

### Step 2: Resolve Parent Issue Node IDs

For each unique parent issue number, look up its GraphQL node ID. Use a single batched query:

```graphql
{
  repository(owner: "FinOps-Open-Cost-and-Usage-Spec", name: "FOCUS_Spec") {
    fr982: issue(number: 982) { id }
    fr1060: issue(number: 1060) { id }
  }
}
```

Build a mapping of `issue_number → node_id` for use in Step 4.

If a parent issue is `N/A (...)`, skip the lookup — there will be no sub-issue relationship for that AI.

### Step 4: Create Issues via GraphQL

For each action item, use the GraphQL `createIssue` mutation. This allows setting the issue type to "Action Item" at creation time (the `gh issue create` CLI does not support `--type`).

**Resolve assignee ID** first — look up the GitHub user's node ID:

```graphql
{ user(login: "{handle}") { id } }
```

**Create the issue:**

```graphql
mutation {
  createIssue(input: {
    repositoryId: "R_kgDOI6Kp7A"
    title: "{title}"
    body: "{body}"
    assigneeIds: ["{user_node_id}"]
    issueTypeId: "IT_kwDOB2KTRs4BgDta"
  }) {
    issue {
      id
      number
      url
    }
  }
}
```

**Body format** — construct the issue body as:

```markdown
**Assignee(s):** @{handle}
**Parent Issue:** #{number}

**Detailed Description**
{description paragraph}

**Definition of Done**
- [ ] {criterion 1}
- [ ] {criterion 2}
```

This matches the format already used by existing `[AI]` issues in the repo (e.g., #1965, #1969).

**Handling edge cases:**
- If `Parent Issue` is `N/A (...)`, omit the parent issue line from the body and include a context line instead (e.g., `**Context:** 1.5 planning`)
- If multiple assignees are listed (comma-separated), resolve each user ID and pass all in `assigneeIds`
- Capture the `issue.id`, `issue.number`, and `issue.url` from the response

### Step 5: Assign Parent Relationships (Sub-Issues)

For each AI that has a numeric parent issue, use the `addSubIssue` mutation to create the real parent-child relationship:

```graphql
mutation {
  addSubIssue(input: {
    issueId: "{parent_node_id}"
    subIssueId: "{new_issue_node_id}"
  }) {
    issue { number }
    subIssue { number }
  }
}
```

Batch these into a single mutation when possible (multiple aliases in one call):

```graphql
mutation {
  sub1: addSubIssue(input: {issueId: "{parent_982_id}", subIssueId: "{issue_1970_id}"}) {
    subIssue { number }
  }
  sub2: addSubIssue(input: {issueId: "{parent_982_id}", subIssueId: "{issue_1971_id}"}) {
    subIssue { number }
  }
}
```

Skip this step for any AI with `N/A` as parent.

Print each result as it completes:
```
Created #1970: [AI] Update PR #1966 to split row 18... (parent: #982, type: Action Item)
Created #1971: [AI] Add suggestion on PR #1966... (parent: #982, type: Action Item)
```

### Step 6: Set Project Fields (Due Date + Task Force + AI Status)

After issue creation, the `github-project-automation` bot automatically adds new issues to the FOCUS WG project. This takes a few seconds. The skill must wait for the project item to appear before setting fields.

**Wait for project assignment:**

Poll each issue's `projectItems` until a FOCUS WG project item appears (max 30 seconds, polling every 5 seconds):

```graphql
{
  repository(owner: "FinOps-Open-Cost-and-Usage-Spec", name: "FOCUS_Spec") {
    issue(number: {N}) {
      projectItems(first: 5) {
        nodes {
          id
          project { id }
        }
      }
    }
  }
}
```

Look for a project item where `project.id == "PVT_kwDOB2KTRs4AcltG"` (FOCUS WG). Extract its `id` as the `itemId`.

**Set Due Date, Task Force, and AI Status** in a single batched mutation per issue:

```graphql
mutation {
  setDueDate: updateProjectV2ItemFieldValue(input: {
    projectId: "PVT_kwDOB2KTRs4AcltG"
    itemId: "{item_id}"
    fieldId: "PVTF_lADOB2KTRs4AcltGzgnLCQI"
    value: { date: "{creation_date + 7 days}" }
  }) {
    projectV2Item { id }
  }
  setTaskForce: updateProjectV2ItemFieldValue(input: {
    projectId: "PVT_kwDOB2KTRs4AcltG"
    itemId: "{item_id}"
    fieldId: "PVTSSF_lADOB2KTRs4AcltGzgSe_EU"
    value: { singleSelectOptionId: "{task_force_option_id}" }
  }) {
    projectV2Item { id }
  }
  setAIStatus: updateProjectV2ItemFieldValue(input: {
    projectId: "PVT_kwDOB2KTRs4AcltG"
    itemId: "{item_id}"
    fieldId: "PVTSSF_lADOB2KTRs4AcltGzgnLF_g"
    value: { singleSelectOptionId: "d6275cd2" }
  }) {
    projectV2Item { id }
  }
}
```

**Task Force mapping** — use the TF number extracted from the analysis file header to look up the option ID from the Constants section. For example, `TF-3` maps to option ID `7a540581`.

**Due date** — check the AI's **Detailed Description** for an explicit target date. Look for patterns like `target: YYYY-MM-DD`, `target: Month DD`, `due date: ...`, or `(target: ...)`. If an explicit date is found, use it. Otherwise, fall back to 7 calendar days from the current date (the day the skill runs). When displaying the summary table (Step 3), show the due date for each AI so the user can verify before creation.

**AI Status** — always set to "Open" (`d6275cd2`) for newly created action items.

**If the `project` token scope is missing**, print a note and skip:
```
Note: Project fields not set — missing project scope. Run: gh auth refresh -s project
```

**If the project item does not appear within 30 seconds**, print a warning and skip:
```
Warning: Project item not found for #{number} after 30s — due date and task force not set. Set manually or re-run.
```

Do not fail the overall workflow if project fields cannot be set.

### Step 7: Update Analysis File

Read the analysis file again and perform find-and-replace for each `#TBD` placeholder.

For each created AI, find the pattern including the surrounding backtick-wrapped title text and parentheses:
```
(`[AI] {exact title} #TBD`)
```
and replace the entire match (including backticks and parentheses) with just the issue number:
```
#{number}
```

For example:
```
Before: * **@rileyjenk**: Upload metadata presentation to Google Drive folder (`[AI] Upload metadata presentation to Google Drive folder for FR #1060 #TBD`)
After:  * **@rileyjenk**: Upload metadata presentation to Google Drive folder #1972
```

GitHub auto-renders `#1972` as a clickable link showing the full issue title on hover, so the redundant title text is no longer needed once the real issue number exists.

Also update the **#TBD Replacement Guide** section — for each entry, replace `[AI] {title} #TBD` with `#{number}` (keeping the guide as a record of what was created).

Write the updated file back to the same path.

### Step 7.5: Post PR Action Comments

For each PR Action parsed in Step 1b, post a comment on the target PR.

**Comment body format:**

```markdown
## TF{#} Action Item - {Meeting Date}

{Description from the PR Action}

**Scope:**
- [ ] {item 1}
- [ ] {item 2}

**Assignee:** @{handle}
```

**General comments** (`Comment Type: general-comment`):

Use the GitHub REST API to post a comment on the PR:

```bash
gh api repos/FinOps-Open-Cost-and-Usage-Spec/FOCUS_Spec/issues/{PR#}/comments \
  -f body="{formatted comment body}"
```

**Review comments** (`Comment Type: review-comment`):

1. Fetch the PR diff to identify the target file and diff position:
   ```bash
   gh api repos/FinOps-Open-Cost-and-Usage-Spec/FOCUS_Spec/pulls/{PR#}
   ```
   Extract the `head.sha` for the `commit_id` parameter.

2. Fetch the PR files to map `File/Line Hint` to an actual file path and diff line:
   ```bash
   gh api repos/FinOps-Open-Cost-and-Usage-Spec/FOCUS_Spec/pulls/{PR#}/files
   ```

3. Use the `File/Line Hint` from the analysis to locate the target area in the diff. Present the proposed comment placement (file, line) for user confirmation before posting.

4. After confirmation, post the review comment:
   ```bash
   gh api repos/FinOps-Open-Cost-and-Usage-Spec/FOCUS_Spec/pulls/{PR#}/comments \
     -f body="{formatted comment body}" \
     -f commit_id="{head_sha}" \
     -f path="{file_path}" \
     -F line={line_number} \
     -f side="RIGHT"
   ```

**Print each result as it completes:**
```
Posted on PR #1966: [PR Action] Address comments and update phrasing (general-comment)
Posted on PR #1966: [PR Action] Split requirement on row 18 (review-comment, effectivecost.md:18)
```

**If no PR Actions exist**, skip this step and note: "No PR Action Items to post."

## ERROR HANDLING

- If `gh` is not authenticated, instruct user to run `gh auth login`
- If the repo is not accessible, report the permission error
- If an assignee handle cannot be resolved (user lookup fails), create the issue without the assignee and flag it
- If `addSubIssue` fails (e.g., parent issue doesn't exist), report the error but do not fail the overall workflow
- If the analysis file has no `#TBD` placeholders, skip Step 7 and note that no updates were needed
- If `project` scope is missing, skip Step 6 and print a note
- If the project item doesn't appear within 30 seconds after issue creation, skip project field updates for that issue and warn
- If a PR Action comment fails to post, report the error (PR number, HTTP status) but continue with remaining items
- If the `File/Line Hint` cannot be matched to a diff position for a review comment, fall back to posting a general comment instead and note the fallback
- If the analysis file has no `### PR Action Items` subsection or it contains "None", skip Step 7.5
